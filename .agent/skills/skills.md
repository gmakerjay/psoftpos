# Dev Discipline — วินัยการพัฒนาซอฟต์แวร์แบบเป็นกลาง

สกิลนี้ไม่ผูกกับภาษา เฟรมเวิร์ก หรือโปรเจกต์ใดโดยเฉพาะ ใช้ได้กับงานพัฒนาซอฟต์แวร์ทุกประเภท เป้าหมายมี 2 อย่าง:

1. **บังคับให้ทำงานบนข้อเท็จจริงที่ตรวจสอบได้จริงเท่านั้น** ไม่เดา ไม่มั่ว ไม่สมมติเอาเอง
2. **ให้กู้คืนสถานะงานได้แม้ context หาย** — โดยพึ่งพาไฟล์บนดิสก์ ไม่ใช่ความจำใน context window

ทั้งสองข้อทำงานร่วมกัน: กฎเหล็กบังคับว่า "ห้ามพูดสิ่งที่ไม่ได้ตรวจสอบ" และไฟล์ PROGRESS.md คือที่เก็บสิ่งที่ตรวจสอบแล้วนั้น เพื่อไม่ให้ต้องพึ่งความจำที่อาจเพี้ยนหรือหายไป

---

## กฎเหล็ก (Iron Rules) — ห้ามฝ่าฝืนไม่ว่ากรณีใด

1. **ห้ามเดา ต้องตรวจสอบก่อนอ้าง** — ก่อนพูดว่าไฟล์/ฟังก์ชัน/config/behavior เป็นแบบไหน ต้องเปิดอ่านของจริง (view / grep / bash) ก่อนเสมอ ถ้ายังไม่ได้เปิดดู ห้ามพูดราวกับว่ารู้แน่ชัด

2. **ห้ามสร้าง API, method, พารามิเตอร์ หรือ package ที่ไม่มีจริง** — ถ้าไม่แน่ใจว่ามีจริงหรือ signature เป็นแบบไหน ให้ตรวจสอบจาก docs/source/package manifest หรือค้นเว็บก่อน ถ้าตรวจสอบไม่ได้ ให้บอกตรงๆ ว่าไม่แน่ใจ แทนที่จะเขียนโค้ดที่ "น่าจะถูก"

3. **ทุกอย่างที่รันได้ ต้องรันจริงก่อนสรุปผล** — ห้ามพูดว่า "น่าจะผ่าน", "ทดสอบแล้ว", "compile ได้" โดยไม่ได้รันจริง ถ้ารันไม่ได้จริงๆ (เช่น ติด network/สภาพแวดล้อม) ให้บอกตรงๆ ว่ายังไม่ได้รัน อย่าสร้างผลลัพธ์สมมติ

4. **แยกข้อเท็จจริงกับสมมติฐานให้ชัดเจนเสมอ** — เวลาต้องตัดสินใจแทนผู้ใช้ (ตั้งชื่อตัวแปร, สมมติ input format, เลือก default) ต้องระบุว่านี่คือ "สมมติฐาน" อย่างชัดเจนในคำตอบหรือ comment ในโค้ด ไม่ปนกับข้อเท็จจริง

5. **ห้ามบอกว่างานเสร็จ/แก้แล้ว ถ้ายังไม่ได้ตรวจสอบจริง** — ต้องเช็ค error, edge case, และผลลัพธ์จริงก่อนสรุปว่า "เรียบร้อย" คำพูดเช่น "แก้บั๊กแล้ว" ต้องแปลว่ารันซ้ำแล้วเห็นผลจริง ไม่ใช่แค่แก้โค้ดแล้วคาดว่าน่าจะโอเค

6. **ไม่แน่ใจ → ถามหรือหยุด ไม่ใช่เดาต่อ** — ถ้าข้อมูลไม่พอสำหรับตัดสินใจอย่างมีเหตุผล ให้ถามคำถามที่ตรงจุดที่สุด 1 ข้อ แทนที่จะเดาแล้วเดินหน้าต่อ

7. **การกระทำที่ทำลายของเดิมต้องแจ้งก่อนเสมอ** — force push, drop table/column, overwrite ไฟล์สำคัญ, ลบข้อมูล ต้องบอกผู้ใช้และรอการยืนยันก่อน (ยกเว้นผู้ใช้สั่งไว้ชัดเจนแล้วล่วงหน้า)

8. **อ้างอิงแหล่งที่มาที่ตรวจสอบได้เสมอ** — เวลาบอกว่า "โค้ด/ไฟล์นี้ทำแบบนี้" ให้ระบุ path หรือบรรทัดจริงที่อ้างอิง ไม่ใช่สรุปจากความจำหรือคาดเดาโครงสร้างโปรเจกต์

9. **ห้ามยัดเหตุผลย้อนหลังให้ดูดีกว่าที่ทำจริง** — ถ้าทำอะไรไปแบบลองผิดลองถูกหรือข้ามขั้นตอนบางอย่าง ให้บอกตามจริง ไม่ต้องเล่าราวกับมีแผนที่สมบูรณ์มาตั้งแต่ต้น

## ระบบกู้คืน Context: ไฟล์ `PROGRESS.md`

ปัญหาที่สกิลนี้แก้: ถ้า context เต็ม ถูก compact หรือเริ่ม session ใหม่ ความจำใน context อาจหายหรือเพี้ยน แล้วอาจ "มั่ว" ต่อจากจุดที่ไม่รู้จริงว่าเกิดอะไรขึ้นมาก่อน วิธีแก้คือ**เขียนสถานะที่ตรวจสอบแล้วลงไฟล์บนดิสก์เสมอ แทนที่จะพึ่งความจำใน context**

### กติกาบังคับ

**ก่อนเริ่มงานพัฒนาใดๆ ในโปรเจกต์** (ไม่ว่าจะเป็น turn แรก หรือ session ใหม่ที่ context ถูกล้างไปแล้ว):

- เช็คว่ามีไฟล์ `PROGRESS.md` อยู่ในโฟลเดอร์โปรเจกต์หรือไม่
  - **ถ้ามี** → เปิดอ่านทั้งไฟล์ก่อนพูดหรือทำอะไรต่อ ห้ามข้าม ห้ามสันนิษฐานว่า "context ยังมีอยู่แล้วไม่ต้องอ่านซ้ำ"
  - **ถ้าไม่มี** → สร้างไฟล์นี้ขึ้นก่อนเริ่มลงมือ

- ถ้าสิ่งที่จำได้ใน context ขัดแย้งกับสิ่งที่เขียนไว้ใน `PROGRESS.md` ให้ **เชื่อไฟล์บนดิสก์เป็นหลัก** เพราะเป็นสิ่งที่ตรวจสอบและบันทึกไว้จริง ไม่ใช่ความจำที่อาจคลาดเคลื่อน

### โครงสร้างไฟล์ (ใช้ template นี้เสมอ)

```markdown
# Progress Log — <ชื่อโปรเจกต์/งาน>
Last updated: <วันที่-เวลา>

## เป้าหมายของงานนี้
<สรุปสั้นๆ ว่าผู้ใช้ต้องการอะไร ผลลัพธ์สุดท้ายคืออะไร>

## สถานะปัจจุบัน
<ทำอะไรไปแล้วบ้าง กำลังทำอะไรอยู่ ณ จุดนี้>

## ข้อเท็จจริงที่ตรวจสอบแล้ว (verified facts)
- <สิ่งที่ยืนยันแล้วด้วยการอ่านไฟล์จริง/รันจริง — ระบุ path หรือคำสั่งที่ใช้ตรวจสอบด้วย>

## สมมติฐานที่ยังไม่ยืนยัน (open assumptions)
- <สิ่งที่สันนิษฐานไว้แต่ยังตรวจสอบไม่ครบ 100% — ต้องระบุไว้ตรงนี้ ห้ามปนกับข้อเท็จจริงด้านบน>

## สิ่งที่ยังไม่เสร็จ / ต้องทำต่อ
- [ ] ...
- [ ] ...

## เสร็จแล้ว (ประวัติ — ห้ามลบ ให้ย้ายมาไว้ตรงนี้แทน)
- [x] ...

## ปัญหา/บั๊กที่รู้อยู่แล้วแต่ยังไม่แก้
- ...

## ไฟล์สำคัญที่เกี่ยวข้อง/แก้ไขแล้ว
- <path> — <เหตุผลที่เกี่ยวข้อง>
```

### กติกาการดูแลไฟล์

- **ห้ามลบประวัติเก่าทิ้ง** — งานที่เสร็จแล้วให้ย้ายไปหัวข้อ "เสร็จแล้ว" แทนการลบ เพื่อให้ตรวจสอบย้อนหลังได้ว่าทำอะไรไปบ้าง
- **อัปเดตทันทีที่มีความคืบหน้าสำคัญ** ไม่ใช่รอให้ผู้ใช้เตือนหรือรอจบงานทั้งหมด โดยเฉพาะก่อนจบ turn ที่มีการเปลี่ยนแปลงสำคัญ
- **แยกช่องข้อเท็จจริง vs สมมติฐานอย่างเคร่งครัด** — ถ้าภายหลังตรวจสอบสมมติฐานได้แล้วว่าจริง ให้ย้ายขึ้นไปช่อง "ข้อเท็จจริงที่ตรวจสอบแล้ว" พร้อมหลักฐาน
- ถ้าโปรเจกต์มีระบบ progress/changelog อยู่แล้ว (เช่น CHANGELOG.md, TODO.md ของโปรเจกต์เอง) ให้ถามผู้ใช้ว่าจะรวมกับของเดิม หรือสร้าง PROGRESS.md แยกไว้เฉพาะสำหรับ Claude ใช้ติดตามงาน

## กฎการ Build & Release โปรเจค (Project Build Rules)
1. **เวอร์ชันคงที่ `1.0.0` เสมอ**: ในการ Build โปรแกรมส่งมอบ ให้ใช้ชื่อเวอร์ชัน `1.0.0` (เช่น `StorePOS_v1.0.0`) เสมอ จนกว่าผู้ใช้จะสั่งเปลี่ยนเวอร์ชันอย่างเป็นทางการ
2. **ปลายทางส่งมอบบน Desktop**: นำไฟล์ที่ Build พร้อมใช้งานแล้วในเครื่องปลายทาง ไปจัดตั้งไว้ในโฟลเดอร์บน Desktop (เช่น `C:\Users\admin\Desktop\StorePOS_v1.0.0`) โดยเปิดโครงสร้างไว้เพื่อให้ผู้ใช้เป็นคน Zip เอง
3. **คอมไพล์เฉพาะตัวเต็ม (Full) และเครื่องมือ KeyGen**: ให้ข้ามการคอมไพล์เวอร์ชันทดลอง 30 วันในการ Build มาตรฐานเพื่อประหยัดเวลา บิวด์เฉพาะตัวโปรแกรมเต็ม (`StorePOS_Full`) และเครื่องมือดูแลระบบ (`Tools/KeyGen.exe`) เสมอ เว้นแต่ผู้ใช้จะระบุสั่งเป็นพิเศษ
4. **กฎการรันของเครื่องมือ KeyGen ในเครื่องปลายทาง**:
   - การสั่งเปิดโปรแกรมอัตโนมัติ (`run_reset_and_launch_activate`) จะต้องสแกนหาและเรียกใช้ตัวโปรแกรมสำเร็จรูป `.exe` (เช่น `StorePOS.exe` หรือ `StorePOS_30DayTrial.exe`) โดยตรง แทนการเรียกหา `main.py`
   - การสแกนหาพาธสิทธิ์เพื่อถอนไลเซนส์แบบอัตโนมัติ จะต้องสแกนหาโฟลเดอร์รุ่นต่างๆ (เช่น `StorePOS_v1.0.0`) บน Desktop และ Documents เสมอ เพื่อให้การปลดล็อก/ลบข้อมูลถอนสิทธิ์ทำได้สมบูรณ์แม้ย้ายเครื่องมือออกไปรันจากจุดอื่น
